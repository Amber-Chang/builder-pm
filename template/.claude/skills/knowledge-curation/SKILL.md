# Knowledge Curation

這是 Planner 的**選用 Skill**，不是自動知識庫，也不是「第二大腦」。目的只有一個：把來源整理成可追溯的候選內容，讓 PM 做決策。未經 PM 明確核准的內容，永遠不是正式事實、需求、PRD 或 SPEC。

## 核心規則

- 先判斷來源類型，再決定閱讀方式與可信度；不要把來源本身當成已驗證事實。
- 優先閱讀正式文件（例如產品政策、現行 PRD / SPEC、API 或法規文件），並記錄文件名稱、版本或日期、章節與連結/路徑。
- 訪談、會議紀錄、客服回報、issue、訊息與程式碼只能作為 evidence（證據材料）；除非已有正式文件或 PM 確認，不得直接改寫成產品承諾。
- 來源互相矛盾時，並列矛盾與影響，不自行裁決；列入待確認問題。
- 只產出候選摘要，不自動升格、不自動寫入正式 `.context/`、PRD 或 SPEC，也不自動 commit。
- 敏感資料（尤其逐字稿、個資、機密商業資訊）不得放進 Git。需要處理時只使用 PM 指定的受控、已 gitignore 的位置；輸出中以最小必要摘錄或去識別化摘要取代逐字內容。
- 不實作 Obsidian、雙向連結、排程、向量資料庫或其他「第二大腦」能力；這些不在本 Skill 範圍內。

## 流程

### 1. 判斷來源類型

先列出收到的來源與分類，至少使用下列分類：

| 類型 | 例子 | 預設用途 |
|------|------|----------|
| 正式文件 | 現行 PRD / SPEC、政策、API / 法規文件 | 了解已批准或具權威性的基準，仍需核對版本與適用範圍 |
| 內部敘事 | 訪談、會議紀錄、客服回報、issue、Slack / email | 發現需求、痛點、例外與待驗證假設 |
| 實作 evidence | 程式碼、測試、設定、執行結果 | 了解目前行為，不代表應保留的產品行為 |
| 外部資料 | 使用者提供的研究、競品資料、公開文章 | 提供背景或假設，需標記來源與時效 |

若來源類型或版本不明，標記為「待確認」，不要猜測。

### 2. 閱讀與記錄正式文件

先讀取與主題最相關的正式文件，再讀敘事或實作 evidence。每個結論都要能回指來源位置；找不到原文位置時，寫「未定位」並降低信心。對現行文件要檢查日期、版本、適用產品與是否已被更新。

### 3. 產出候選摘要

使用下列結構，維持「來源原文/觀察」與「推論/候選」分開：

```markdown
## 候選摘要：<主題>

### 來源清單
- <來源類型>：<名稱、版本/日期、章節或路徑>

### 已觀察到的 evidence
- <可回指來源的觀察>（來源：<位置>）

### 候選需求 / 規則
- <候選內容>（依據：<來源位置>；狀態：candidate）

### 信心
- <高 / 中 / 低>：<為什麼；哪些部分仍可能改變>

### 待 PM 確認
- <要 PM 決定或補資料的問題>

### 不納入的內容
- <推測、衝突或超出範圍的內容，以及原因>
```

「高信心」只表示來源明確、一致且可追溯，不等於已獲 PM 批准；「中信心」表示有部分支持但仍有缺口；「低信心」表示主要是推論、proxy（代理訊號）或來源衝突。不能以信心等級取代核准。

### 4. 等待 PM 核准

交付候選摘要後停止，不直接建立或更新 PRD / SPEC。請 PM 對每一項候選明確選擇：

- 核准：可協助寫入指定 PRD / SPEC，並保留來源與決策紀錄。
- 修改後核准：先依 PM 指示修訂，再重新交付確認。
- 不採用：保留為 rejected（不採用）決策，不移入正式文件。
- 需補資料：列出具體缺口，不能用猜測補齊。

只有在 PM 明確核准後，才可協助建立或更新指定的 PRD / SPEC；若需求邏輯仍不確定，加入 `// TODO: [NEED-REVIEW] 請工程師確認`，並把技術決策交給工程師。

## 完成檢查

- [ ] 所有來源已分類，正式文件已先閱讀並記錄版本/位置。
- [ ] evidence、候選、推論與已核准內容分開。
- [ ] 每項候選都有信心等級、依據與待確認問題。
- [ ] 已明確等待 PM 核准，沒有自動更新正式文件或 commit。
- [ ] 敏感逐字稿與其他本機 AI 產物未進 Git。
- [ ] 沒有新增 Obsidian、雙向連結、排程或第二大腦實作。
